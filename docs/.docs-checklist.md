# Documentation Maintenance Checklist

This checklist helps ensure documentation stays up-to-date with code changes.

## When to Update Documentation

### Code Changes

- [ ] **New Command Added** → Update `commands.md` and `getting-started.md`
- [ ] **Command Modified** → Update `commands.md`
- [ ] **New Feature** → Update relevant docs + `CHANGELOG.md`
- [ ] **Bug Fix** → Update `CHANGELOG.md`
- [ ] **Architecture Change** → Update `architecture.md`
- [ ] **UI Change** → Update `ui-guide.md`
- [ ] **Configuration Change** → Update `getting-started.md` and `architecture.md`
- [ ] **Dependency Added/Updated** → Update `architecture.md` and `development.md`

### Release Process

- [ ] Update `CHANGELOG.md` with version and date
- [ ] Update version badges in `README.md`
- [ ] Review all documentation for accuracy
- [ ] Check all code examples still work
- [ ] Verify all links are valid
- [ ] Update roadmap in `README.md`

## Documentation Files Reference

### README.md (Root)
**Purpose**: Project overview and quick start
**Update When**: 
- Project description changes
- Quick start process changes
- Major features added/removed
- Roadmap updates

### docs/README.md
**Purpose**: Documentation index
**Update When**:
- New documentation files added
- Documentation structure changes
- Quick links need updating

### docs/getting-started.md
**Purpose**: Installation and basic usage guide
**Update When**:
- Installation process changes
- Configuration options change
- Basic usage patterns change
- Troubleshooting steps added

### docs/commands.md
**Purpose**: Complete command reference
**Update When**:
- New commands added
- Command flags/options change
- Command behavior changes
- Examples need updating

### docs/architecture.md
**Purpose**: System design and technical decisions
**Update When**:
- Architecture changes
- New dependencies added
- Design patterns change
- Technology stack updates

### docs/ui-guide.md
**Purpose**: Bubble Tea TUI implementation details
**Update When**:
- UI components added/modified
- Model structure changes
- New keyboard shortcuts added
- View rendering logic changes

### docs/development.md
**Purpose**: Developer guide for contributors
**Update When**:
- Development setup changes
- Build process changes
- Testing procedures change
- Code style guidelines update
- New development tools added

### docs/CHANGELOG.md
**Purpose**: Version history and changes
**Update When**:
- ANY code change (add to Unreleased section)
- New version released (move Unreleased to versioned section)
- Breaking changes introduced

## Quick Update Guide

### Adding a New Command

1. Implement command in `cmd/`
2. Update `docs/commands.md`:
   - Add command to table of contents
   - Add command section with usage, description, examples
3. Update `docs/getting-started.md`:
   - Add to usage examples if it's a common command
4. Update `docs/CHANGELOG.md`:
   - Add to "Added" section under [Unreleased]
5. Update root `README.md`:
   - Add to features list if significant

### Modifying the TUI

1. Make changes to model/update/view in `cmd/root.go`
2. Update `docs/ui-guide.md`:
   - Update model structure if changed
   - Update keyboard controls if changed
   - Update view rendering if changed
3. Update `docs/getting-started.md`:
   - Update navigation section if controls changed
4. Update `docs/CHANGELOG.md`:
   - Add to "Changed" section under [Unreleased]

### Adding a Configuration Option

1. Add configuration in code
2. Update `docs/getting-started.md`:
   - Add to configuration section
   - Add example usage
3. Update `docs/architecture.md`:
   - Add to configuration management section
4. Update `docs/CHANGELOG.md`:
   - Add to "Added" section under [Unreleased]

### Fixing a Bug

1. Fix the bug
2. Update `docs/CHANGELOG.md`:
   - Add to "Fixed" section under [Unreleased]
3. If bug was documented in troubleshooting:
   - Update or remove from `docs/getting-started.md`

## Documentation Quality Checklist

Before committing documentation changes:

- [ ] All code examples are tested and work
- [ ] All links are valid (internal and external)
- [ ] Markdown formatting is correct
- [ ] Spelling and grammar checked
- [ ] Technical accuracy verified
- [ ] Examples are up-to-date with current code
- [ ] Screenshots/diagrams updated if needed
- [ ] Consistent terminology used throughout
- [ ] All new features documented
- [ ] CHANGELOG.md updated

## Documentation Review Schedule

### Weekly
- [ ] Review recent code changes
- [ ] Update CHANGELOG.md
- [ ] Check for outdated examples

### Monthly
- [ ] Full documentation review
- [ ] Verify all links
- [ ] Update version information
- [ ] Review and update roadmap

### Before Each Release
- [ ] Complete documentation audit
- [ ] Update all version references
- [ ] Finalize CHANGELOG.md
- [ ] Review getting started guide
- [ ] Test all code examples
- [ ] Update README.md roadmap

## Common Documentation Patterns

### Command Documentation Template

```markdown
### command-name

Brief description.

#### Usage

\`\`\`bash
qnq command-name [flags]
\`\`\`

#### Description

Detailed description of what the command does.

#### Flags

| Flag | Short | Description | Default |
|------|-------|-------------|---------|
| --flag | -f | Description | value |

#### Examples

\`\`\`bash
# Example 1
qnq command-name --flag value

# Example 2
qnq command-name -f value
\`\`\`
```

### Changelog Entry Template

```markdown
### Added
- New feature description with details

### Changed
- Changed feature description with migration notes if needed

### Fixed
- Bug fix description with issue reference if available
```
